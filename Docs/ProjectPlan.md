# WinChocolate — Build Plan

## Summary

WinChocolate is an AppKit-shaped Swift framework for Windows. Apple AppKit API compatibility is the project's primary goal: apps are built with the Apple API and look like Windows apps. The near-term focus is a dependable classic Win32 backend with enough Cocoa/AppKit compatibility to port small apps, while preserving a path toward modern Windows visuals, richer layout, and deeper Foundation parity. A Linux sibling, **LinChocolate**, follows once Windows is going — same Apple API in, native Linux look out; it has its own plan in `Docs/LinChocolatePlan.md`.

This plan is the high-level project tracker. `CONTROL_PARITY.md` remains the detailed control-by-control map, and `Architecture.md` remains the design overview.

## Project Goals

1. **PRIMARY GOAL — Apple AppKit API compatibility.** WinChocolate exists so that almost all Mac AppKit programs build and run on Windows by swapping `import AppKit` for `import WinChocolate`. Application code is written against the Apple API — AppKit names, types, delegates, and behavior — while the controls render as native Windows controls. For most controls the app should *look* like a Windows app but be *built* with the Apple API. When any design decision conflicts with this goal, Apple API compatibility wins. (Deliberate exception: toolbars keep the Apple look and feel — see Phase 6.)
2. **Selectable presentation.** Applications should eventually be able to select either the current classic Win32 look or the more modern Windows look through one switch, with no other application code changes. The classic look stays fully supported; the modern look becomes the default later in the plan once Phase 8 reaches parity.
3. **Real native backing.** AppKit-shaped API in front, honest Windows implementation behind a narrow backend boundary, kept testable through the in-memory backend and contract tests.

## Working method — milestones

Each phase is a milestone, and work always drives toward completing the **current** milestone before advancing. A milestone is never left blocked by a later one: if a current-phase item depends on an item scheduled for a future phase, that future item is **pulled into the current phase and done now**, then renumbered here with percentages recomputed. (Already applied: `Timer` and `FileManager` were pulled from Phase 7 into Phase 3 to unblock `NSDocument` autosave/close — see 3.14 and 3.15.)

**Scope comes from the AppKit surface itself, not from the test apps.** The goal is that most AppKit programs build and run (at least their UI); the Phase 11 apps only demonstrate that. When any work reveals a missing AppKit capability, it is added here as a first-class item in the phase it belongs to — never deferred "until a test app needs it."

**Skipped work is moved, never dropped.** Each candidate is triaged individually. The rule of thumb: **cosmetic/polish items may be resequenced to a later cleanup phase (Phase 10) so long as the current implementation works and isn't hideous**; functional gaps stay in the current milestone unless explicitly triaged out. Anything deferred is relocated to a tracked item with a "(Moved from X — sequenced to cleanup, not skipped)" note and the percentages recomputed, so nothing is forgotten. (Applied: NSSearchField search chrome → 10.8; sheet animation/window-modal → 10.9.)

## Phase 15 — NIB / XIB Support ⏳ 0%

*(Added 2026-07-11 at user direction. This **reverses the earlier "nib loading is out of scope" stance** noted in 12.1 — bindings stay deferred, but nib/xib loading is now in scope as its own phase.)* Load Interface Builder documents so nib-based AppKit apps run without hand-writing their UI. This is a **subsystem, not a name gap**, and it lands on top of the control set (Phases 3/5/6, done), Auto Layout (Phase 9 — xibs embed constraints), and menus/windows.

**The two hard design questions to settle first:**
1. **`.xib` (source XML) vs compiled `.nib` (binary keyed-archive).** Apple's toolchain compiles `.xib` → `.nib` at build time and apps ship the opaque binary `.nib` (an `NSKeyedArchiver` plist we have no unarchiver for). The tractable path on Windows is to **parse the `.xib` XML document directly** (it is human-readable and stable) and skip the compile step — developer-friendly and avoids reimplementing the keyed-unarchiver. Supporting shipped `.nib`s would additionally need a keyed-archive reader. **Lead with `.xib`.**
2. **Outlet/action wiring needs KVC/reflection, which Swift-on-Windows lacks** — the *same* infrastructure gap that defers Cocoa bindings (12.1). Setting `@IBOutlet var button: NSButton!` by string name, or invoking an `@IBAction` selector, needs a runtime we don't have. **Realistic first slice:** parse + instantiate the object graph and hand the app the **top-level objects + a lookup of identified objects** so it wires outlets itself (or via a registration closure) — full automatic `@IBOutlet` binding is a later slice gated on a KVC-like layer.

### Cross-Foundation Codable interchange (raised 2026-07-07; UUID decided 2026-07-07)

> **UUID decision made (Bobby, 2026-07-07): WinFoundation now owns `UUID: Codable`.** `WinFoundation/Sources/WinFoundation/UUID.swift` declares the conformance directly, matching Apple Foundation's exact form — a single-value container holding the uppercase `uuidString` — so JSON is byte-identical across the two Foundations (`testWinFoundationUUIDCodableMatchesAppleForm`: encodes the uppercase string, round-trips, accepts Apple's lowercase form on decode, throws on malformed input). Uses only the stdlib `Codable`, so no Foundation dependency is added. `Date`/`Data` are **not** part of this decision and stay as they were.
>
> **⚠️ DOWNSTREAM ACTION REQUIRED (WinSwiftData):** WinSwiftData currently carries its own **retroactive `extension UUID: Codable`** (the "bridge conformance"). Now that WinFoundation vends the conformance, that retroactive one is a **duplicate/conflicting conformance** and will fail the WinSwiftData build (`redundant conformance of 'UUID' to protocol 'Codable'`). WinSwiftData must **delete its retroactive UUID Codable conformance** and rely on WinFoundation's. This is a coordinated change: it lands here (additive to WinFoundation) but the downstream must drop its shadow before its next clean build. `Date`/`Data` retroactive conformances are unaffected — keep those.



The WinSwiftData/WinSwiftUI session flagged whether JSON its models encode on a Windows/WinFoundation build could round-trip with a Mac/real-Foundation build. The comparison of the three bridged value types against Apple Foundation's `JSONEncoder` defaults:

| Type | Apple Foundation JSON form | Bridge / WinFoundation form | Interchangeable? |
|------|----------------------------|-----------------------------|:----------------:|
| `UUID` | string, uppercase `uuidString` (e.g. `E621E1F8-…`) | string, uppercase `uuidString` — **now WinFoundation-owned `Codable`** (single-value container, byte-identical to Apple) | ✅ Yes |
| `Date` | `Double` seconds since **2001** (`timeIntervalSinceReferenceDate`) | `Double` seconds since **1970** | ⚠️ Different epoch |
| `Data` | base64 **string** (`JSONEncoder` default) | `[UInt8]` **array** | ⚠️ Different form |

`UUID` matches Apple both ways and is now vended by WinFoundation itself (see the decision note above; the previously-relied-on retroactive downstream conformance must be removed). `Date`/`Data` **deliberately** follow WinSwiftData's own storage conventions instead (top-level bridge persists `Date` as `.timestamp` = seconds-since-1970 and `Data` as `.bytes`), so the nested-`Codable` path stays internally consistent with that store. Tradeoff: a nested `Date`/`Data` inside a `.json` attribute is **not** byte-for-byte interchangeable with a real-Foundation build.

**Decision needed (Bobby):** does cross-Foundation JSON interchange matter — e.g. a model file shared between a Mac build and the Windows build?
- **If yes:** switch the bridge to Apple's exact conventions — `Date` → since-2001 `Double`, `Data` → base64 `String`. (A WinFoundation concern if we want `WinFoundation`'s own `Date`/`Data` `Codable` to match Apple; otherwise the choice lives in WinSwiftData's coder.)
- **If Windows-only:** current forms are cleaner and internally consistent — leave them.

Default stance until decided: **leave as-is** (WinSwiftData is the only consumer and its store is Windows-only), and revisit if a genuine Mac↔Windows shared-file scenario appears.

**Handling rule:** treat these like the WinSwiftUI stability contract (Phase 7 intro) — additive, prioritized by how much they simplify the consumer, and closed only when the AppKit-shaped API is real on our side. When one is resolved, record the resolving API here and (optionally) notify the source session through the user.
