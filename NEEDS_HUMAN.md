# Human Review Checklist

| Item | Reason | Reviewed | Human Accepted |
|---|---|---:|---:|
| WinFoundation downstream API freeze (WinSwiftData) | The external WinSwiftData project consumes the `WinFoundation` library product via a read-only path dependency. Since 7.10 the package is the standalone `Code/WinChocolate/WinFoundation/` (WinSwiftData must re-point its path dependency there — one line; coordinate via the user). Keep stable: that package location, the `WinFoundation` product name, and `Date.timeIntervalSince1970`/`Date(timeIntervalSince1970:)`, `UUID.uuidString`/`UUID(uuidString:)`, `Data.array`/`Data(_ sequence:)`. Renames/removals need coordination through the user. | [ ] | [ ] |
| Win32 native backend FFI | The backend now uses manual User32/Gdi32 declarations because this ARM64 Swift toolchain cannot import `WinSDK`; the declarations should be reviewed before broadening the API surface. | [ ] | [ ] |
| Keyboard modifiers on real hardware | VM and remote keyboard layers can remap Command/Windows/Alt into Control or system shortcuts before WinChocolate sees them; modifier behavior should be checked on a real Windows PC. | [ ] | [ ] |
| Alert focus recovery after native MessageBoxW | After dismissing the demo `NSAlert`, keyboard focus can stop moving until the user clicks back in the window. `NSAlert.runModal()` now restores AppKit-side key/main window and first responder, but the classic Win32 `MessageBoxW` path still appears to need a native activation/focus-message fix or a custom alert dialog backend. | [ ] | [ ] |
| Foundation toolchain canary | The current Windows ARM64 Swift toolchain cannot compile real `import Foundation`; Windows builds use `WinFoundation` through `USE_WIN_FOUNDATION`. When a new Swift toolchain is installed, run the canary in `FOUNDATION_SHIMS.md` and migrate shims back to real Foundation if it passes. | [ ] | [ ] |
| WinFoundation URL compatibility | `URL` is now the first bridge type exposed through `NSPathControl` and likely future panels/resources/document APIs. Its API should be reviewed against common Foundation `URL` usage before deeper file-related AppKit surfaces are added. | [ ] | [ ] |
| Win32 backend file split (Native/Win32/) | The backend was split from one ~3,550-line file into 14 files under `Sources/WinChocolate/Native/Win32/` (FFI declarations, core class, message dispatch, and per-control extensions). This was a pure move: ~340 FFI symbols and ~75 class members went from file-private to internal so extensions can reach them. Review the grouping and the widened internal surface. | [ ] | [ ] |
| NSToolbar.swift size (~1,050 lines) | Customization UI was extracted to `NSToolbarCustomizationPanel.swift`, but the file still holds the toolbar model, the composed `NSToolbarView` renderer, composite item views, and `NSToolbarItem`. Consider splitting the renderer and item model into their own files. | [ ] | [ ] |
| Demo main.swift size (~1,900 lines) | The demo is a single top-level file exercising every control. Consider splitting into per-page setup files if it keeps growing. | [ ] | [ ] |
| Native window frame semantics | `NSWindow` frames are now treated as AppKit content-rect sizes: the Win32 backend grows the outer window with `AdjustWindowRectEx` at creation and on `setFrame`. Confirm this matches expectations for apps that position windows precisely, and decide whether `frame.origin` should also be content-based. | [ ] | [ ] |
| LinChocolate: menu bar excluded from popover-dismiss fallback | Regression fix: the outside-click/window-deactivate dismissal walk (`popdownVisiblePopovers`) was recursing into the `GtkPopoverMenuBar` and popping down the open menu. On non-composited XQuartz, opening a menu popover deactivates the window (separate X surface), firing `notify::is-active`, which closed the just-opened menu — so File/Help menus appeared not to work. Fixed by skipping `GtkPopoverMenuBar`/`…BarItem` subtrees in the walk (the menu system itself was always fine — verified via F10→Return firing "Reset Counter" headlessly). Trade-off: our fallback no longer dismisses menu-bar menus on far-outside clicks over XQuartz (GTK-native dismissal + Escape + clicking another top-level item still close them). Could not verify the mouse-open visual headlessly — Xvfb doesn't render GtkPopoverMenuBar menus. **Needs a mouse test on real XQuartz**: click File/Help and confirm the menu drops down and its items fire. | [ ] | [ ] |
| LinChocolate: popover dismissal over XQuartz | GTK popovers don't auto-dismiss on outside click over non-composited XQuartz (the pointer grab is ineffective). Fallbacks now in place: capture-phase in-window click handler **plus** window `is-active`-loss popdown (covers clicks on the desktop/other apps). Also fixed: the dismissal walk was popping down `GtkPopoverMenuBar`/`…BarItem` (not popovers — 3 Gtk-CRITICALs per click, now exact-type matched), and modal alerts are non-modal on non-composited displays so a failed window map can't input-lock the app. Verified under Xvfb (0 CRITICALs, alert lifecycle clean); **needs a re-test on real XQuartz**. | [ ] | [ ] |
| LinChocolate: NSPopover / dropdown dismissal over XQuartz | ✅ Resolved. `NSPopover` and the combo-box dropdown were being popped down the instant they opened by a `notify::is-active` fallback handler (opening an autohide popover deactivates the toplevel on non-composited X11). That handler is removed; the capture-phase click handler still dismisses on in-window outside clicks. **Verified rendering on Xvfb** (both the Fruit dropdown and the "Show Popover" popover now open and stay open). Worth a quick confirm on real XQuartz that clicking elsewhere still dismisses them cleanly. | [ ] | [ ] |
| LinChocolate: live drag-and-drop gesture over XQuartz | The DnD data path (`GtkDragSource`→`GtkDropTarget`, string via `GdkContentProvider`) is contract-tested through the exact registered callbacks, and copy/paste is verified live over GTK. The actual click-drag-release gesture could not be verified headlessly — synthetic Xvfb XTEST events don't reliably trip GTK4's drag threshold/grab. **Needs a manual mouse test on real XQuartz**: on the demo's "Drag & Drop" tab, drag the "Drag me →" chip onto the "Drop here" well and confirm it shows "🍫 dropped payload". | [ ] | [ ] |
